## 🎯 개요

Spring Boot 애플리케이션을 실행하는 방법은 크게 3가지가 있습니다:
1. **systemctl (systemd 서비스)** - 현재 사용 중
2. **Gradle bootRun** - 개발/테스트용
3. **JAR 파일 실행** - 프로덕션 권장

---

## 1. 🔧 systemctl (systemd 서비스) - 현재 방식

### 특징
- **자동 재시작**: 서비스가 비정상 종료되면 자동으로 재시작
- **시스템 부팅 시 자동 시작**: `systemctl enable`로 설정 가능
- **로그 관리**: `journalctl`로 통합 로그 관리
- **서비스 관리**: 표준 Linux 서비스 관리 방식
- **권한 관리**: 특정 사용자로 실행 가능

### 장점
✅ 서버 재부팅 시 자동 시작  
✅ 프로세스 크래시 시 자동 복구  
✅ 표준화된 서비스 관리  
✅ 로그 통합 관리 (journalctl)  
✅ 서비스 상태 모니터링 용이  

### 단점
❌ sudo 권한 필요 (서비스 시작/중지 시)  
❌ 빌드가 포함되지 않음 (수동 빌드 필요)  
❌ Gradle 데몬 프로세스가 함께 실행됨  

### 사용 방법
```bash
# 서비스 시작
sudo systemctl start kalisco-service

# 서비스 중지
sudo systemctl stop kalisco-service

# 서비스 재시작
sudo systemctl restart kalisco-service

# 서비스 상태 확인
sudo systemctl status kalisco-service

# 로그 확인
sudo journalctl -u kalisco-service -f

# 자동 시작 활성화
sudo systemctl enable kalisco-service
```

### 현재 설정
- **서비스 파일**: `/etc/systemd/system/kalisco-service.service`
- **실행 명령**: `gradlew bootRun`
- **사용자**: natus-server
- **자동 재시작**: 활성화 (Restart=always)

## 2. 🛠️ Gradle bootRun - 개발/테스트용

### 특징
- **빌드 포함**: 실행 시 자동으로 컴파일 및 빌드
- **핫 리로드**: 개발 중 코드 변경 시 빠른 재시작
- **디버깅 용이**: IDE에서 직접 실행 가능
- **Gradle 데몬**: Gradle 데몬 프로세스가 함께 실행

### 장점
✅ 빌드와 실행이 한 번에  
✅ 개발 중 빠른 반복 작업  
✅ IDE 통합 용이  
✅ 디버깅 모드 실행 가능  

### 단점
❌ 프로덕션 환경에 부적합  
❌ Gradle 데몬이 메모리 사용  
❌ 자동 재시작 기능 없음  
❌ 서버 재부팅 시 수동 시작 필요  

### 사용 방법
```bash
# 개발 서버 실행 (포그라운드)
cd /var/www/html/project/kalisco
./gradlew bootRun

# 백그라운드 실행
cd /var/www/html/project/kalisco
nohup ./gradlew bootRun > logs/kalisco.log 2>&1 &

# 특정 포트로 실행
./gradlew bootRun --args='--server.port=9091'

# 프로세스 종료
pkill -f "gradle.*kalisco"
```

### 실행 예시
```bash
# 1. 프로세스 종료
pkill -f "gradle.*kalisco" && pkill -f "java.*9091"

# 2. 빌드 및 실행
cd /var/www/html/project/kalisco
./gradlew clean build -x test
nohup ./gradlew bootRun > logs/kalisco.log 2>&1 &

# 3. 상태 확인
sleep 5 && netstat -tlnp | grep 9091
```

## 3. 📦 JAR 파일 실행 - 프로덕션 권장

### 특징
- **독립 실행**: JAR 파일 하나로 모든 의존성 포함
- **빠른 시작**: Gradle 없이 직접 Java로 실행
- **메모리 효율**: Gradle 데몬 프로세스 불필요
- **배포 용이**: JAR 파일만 복사하면 실행 가능

### 장점
✅ 프로덕션 환경에 최적화  
✅ 빠른 시작 시간  
✅ 메모리 사용량 최소화  
✅ 배포 및 이식 용이  
✅ Gradle 의존성 없음  

### 단점
❌ 코드 변경 시 재빌드 필요  
❌ 빌드 시간 소요  
❌ JAR 파일 크기가 큼 (의존성 포함)  

### JAR 파일 생성
```bash
# JAR 파일 빌드
cd /var/www/html/project/kalisco
./gradlew clean bootJar

# 생성된 JAR 파일 위치
ls -lh build/libs/kalisco-0.0.1-SNAPSHOT.jar
```

### JAR 파일 실행 방법

#### 방법 1: 직접 실행
```bash
# JAR 파일 실행
cd /var/www/html/project/kalisco
java -jar build/libs/kalisco-0.0.1-SNAPSHOT.jar

# 백그라운드 실행
nohup java -jar build/libs/kalisco-0.0.1-SNAPSHOT.jar > logs/kalisco.log 2>&1 &

# 특정 포트로 실행
java -jar build/libs/kalisco-0.0.1-SNAPSHOT.jar --server.port=9091
```

#### 방법 2: systemd 서비스로 실행 (JAR 파일 사용)
```bash
# 서비스 파일 수정
sudo nano /etc/systemd/system/kalisco-service.service
```

서비스 파일 내용:
```ini
[Unit]
Description=Kalisco Project Service (Spring Boot)
After=network.target postgresql.service

[Service]
Type=simple
User=natus-server
Group=natus-server
WorkingDirectory=/var/www/html/project/kalisco
ExecStart=/usr/lib/jvm/java-17-openjdk-amd64/bin/java -jar /var/www/html/project/kalisco/build/libs/kalisco-0.0.1-SNAPSHOT.jar
Restart=always
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

#### 방법 3: 전용 스크립트 사용
```bash
# JAR 파일 기반 시작 스크립트
cd /var/www/html/project/kalisco
./start_kalisco_jar.sh

# JAR 파일 기반 종료 스크립트
./stop_kalisco_jar.sh
```

## 🔄 현재 설정 (systemctl + Gradle bootRun)

현재는 **systemctl + Gradle bootRun** 방식을 사용하고 있습니다.

### 현재 방식의 특징
- ✅ 자동 재시작 기능
- ✅ 서버 재부팅 시 자동 시작
- ⚠️ Gradle 데몬이 메모리 사용
- ⚠️ 빌드가 서비스에 포함되지 않음

### 개선 제안: JAR 파일 + systemd

더 효율적인 프로덕션 환경을 위해서는 **JAR 파일 + systemd** 방식으로 변경하는 것을 권장합니다.

#### 변경 절차
1. JAR 파일 빌드
2. systemd 서비스 파일 수정 (ExecStart를 JAR 실행으로 변경)
3. 서비스 재시작
